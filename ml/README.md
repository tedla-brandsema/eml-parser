# ML Subproject

This directory holds the first runnable Python ML scaffold for `eml-parser`.

Its current purpose is narrow and deliberate:

- learn **equivalence-family membership** from sampled data,
- treat that as the first ML proxy for interchangeable EML laws,
- stay small enough to run comfortably on desktop hardware,
- and avoid pretending we already have full symbolic regression or snippet
  assembly.

## Current Task

The first task is `family_match`.

- input: a set of sampled observations from one hidden law
- target: the equivalence-family ID of that law
- objective: learn that different family members can map to the same law-level
  target, rather than optimizing for exact-tree identity

This is intentionally the first ML step because it stays aligned with the real
research direction:

- family structure before exact tree recovery,
- equivalence awareness before assembly,
- and data-driven family inference before larger combinatorics.

## Data Sources

Two dataset paths are supported.

### 1. Mock Family Data

The scaffold is runnable immediately using an in-memory mock generator in
`ml/datasets/mock_family_match.py`.

This generator exists only to make the ML stack executable before the Go-side
equivalence-family corpora exist. It mirrors the future artifact contract
conceptually:

- each sample belongs to one family,
- families may have multiple member variants,
- the model predicts family identity from sampled point sets rather than from
  symbolic strings.

### 2. Go-Generated Artifact Path

The scaffold also includes an artifact-aware loader in
`ml/datasets/artifact_family_match.py`.

That loader reads JSON corpora under `artifacts/equivalence/` generated by the
Go core. This is now the preferred path for `family_match`. The mock generator
remains available as a smoke-test fallback.

The directory may also contain richer `.family.json` equivalence-family corpora.
Those are generated for later paired/grouped and snippet-level tasks. The
current `family_match` loader still targets the simpler anchor-artifact files.

## Models

The first model family mirrors the useful simplicity of `quotient-collapse`:

- `baseline`
  - order-sensitive flattened input baseline
- `set_encoder`
  - DeepSets-style set model over sampled points
- `set_encoder_plus_stats`
  - set encoder plus explicit summary statistics
- `oracle`
  - receives true family-level oracle features and defines an upper bound

## Environment

This subproject uses the same general style as `quotient-collapse`:

- Conda
- PyTorch
- small synthetic datasets
- plain `train.py` and `eval.py`

Create the environment with:

```bash
conda env create -f ml/env.yaml
conda activate eml-ml
```

## Running

Train all first-stage models on the mock family-match task:

```bash
python -m ml.train
```

Train on real Go-generated family artifacts instead:

```bash
go run ./cmd/emltool gen-family-artifacts
python -m ml.train --artifact-dir artifacts/equivalence
```

Evaluate all trained models and write a CSV summary:

```bash
python -m ml.eval
```

Evaluate the artifact-backed path:

```bash
python -m ml.eval --artifact-dir artifacts/equivalence
```

Outputs are written to:

```text
ml/checkpoints/
ml/results/
```

## Metrics

The first scaffold measures:

- `family_accuracy`
- `top3_family_accuracy`
- `permutation_sensitivity`
- `oracle_gap`

These are chosen to keep attention on the actual goal:

- learning family-level interchangeability from sampled data,
- not just obtaining one scalar accuracy number.

## Deliberately Deferred

This scaffold does not yet implement:

- snippet retrieval,
- assembly experiments,
- plotting or paper-facing figures.

The final multi-member equivalence-family corpus format still belongs to later
items in `TODO.md`. The current artifact path is intentionally minimal: one
anchor family artifact per generated law, with deterministic sampled point sets
and provenance.
