# ML And Combinatorics Strategy

## Summary

The project should now treat symbolic regression over EML as a two-stage problem:

1. data to candidate trees or candidate snippets,
2. candidate trees or snippets to equivalence classes and larger assemblies.

The central research claim is no longer only that EML can be searched. It is that EML may support a useful combinatoric regime where:

- different raw trees can explain the same sampled data,
- equivalent subtrees can be interchanged inside larger constructions,
- and partial laws can be scientifically useful even when no full global law is recovered.

This changes the role of both search and ML.

## Monorepo Split

The repository should evolve as a monorepo with two main implementation areas.

### Go Symbolic Core

The Go portion should remain responsible for deterministic symbolic work:

- raw EML parser and AST,
- concept dictionary and expansion,
- evaluation backends,
- controlled normalization,
- synthetic dataset generation,
- equivalence-family generation,
- experiment specs and result artifacts.

This side should be the source of truth for artifact generation.

The Go code remains at repository root for now. The monorepo split is created
around it rather than by moving it into a `/go` subtree.

### Python ML Subproject

The Python `ml/` subproject should handle statistical learning work:

- model training,
- representation learning over tree families,
- snippet discovery from data,
- equivalence-aware ranking or generation,
- assembly experiments from partial trees,
- evaluation against corpora generated by the Go core.

The Python side should consume generated artifacts rather than reimplement symbolic semantics ad hoc.

The reserved on-disk location for that work is `ml/`.

## Synthetic Data First

Synthetic data should now be treated as a first-class research instrument.

The immediate goal is not to fit arbitrary real-world datasets directly. It is to generate controlled datasets where we know:

- the originating tree,
- equivalent alternative trees when known,
- sample domains,
- partial substructures embedded in larger targets,
- and the exact experimental conditions under which recovery should or should not be possible.

This gives the project a clean training and evaluation substrate before noisier real data is introduced.

The repository should distinguish:

- `experiments/` for oracle experiment specs and run artifacts,
- `artifacts/` for shared corpora generated by Go and consumed by `ml/`.

## Equivalence Families

The project should explicitly construct families of trees that are related in different ways.

At minimum, distinguish:

- exact same raw tree,
- normalized same raw tree,
- numerically equivalent on sampled domains,
- known concept-level equivalence,
- contextual subtree equivalence inside a larger tree.

The point is not to collapse these distinctions too early. The point is to study them.

## No Premature Normalization

Normalization remains useful, but it should not be mistaken for identity.

Use normalization for:

- hygiene,
- stable serialization,
- conservative deduplication,
- controlled comparisons.

Do not assume that normalization exhausts the interesting equivalence structure of EML.

In particular, search and ML experiments should be allowed to observe different trees that:

- fit the same sampled data,
- differ structurally,
- and may later turn out to be equivalent.

## Partial Laws And Snippets

The project should stop treating full-formula recovery as the only valuable outcome.

A useful system should also be able to:

- recover EML snippets that explain part of a dataset,
- identify multiple candidate snippets for overlapping regions,
- test whether snippets can be composed into a larger coherent tree,
- preserve partial recoveries when full assembly fails.

This is the intended human-AI collaboration model:

- the system proposes local laws and compatible fragments,
- humans inspect, reinterpret, and bridge remaining gaps.

## Equivalence-Aware Exposure

The future ML layer should be exposed deliberately to training data in which:

- wildly different trees yield the same outputs,
- equivalent subtrees can replace each other,
- and larger expressions remain valid under local substitutions.

This is not accidental data augmentation. It is the core training signal for substitutional flexibility.

The model should learn that:

- one dataset may admit multiple valid decompositions,
- one subtree can sometimes be swapped for another,
- and global coherence can emerge from locally interchangeable pieces.

## Immediate Priorities

The next implementation priorities should be:

1. formalize the monorepo split in docs and repository layout,
2. define artifact contracts between Go and Python,
3. generate synthetic datasets from raw trees and concept expansions,
4. generate paired or grouped equivalence families,
5. define snippet-level datasets and partial-fit evaluation,
6. scaffold the Python ML subproject against those artifacts.

Search improvement remains useful, but it is no longer the sole center of gravity.

The source of truth for ownership and read/write direction is
`.docs/artifact-contract.md`.
