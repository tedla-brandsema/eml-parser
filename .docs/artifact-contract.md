# Artifact Contract

## Summary

This document defines the repository-level contract between the Go symbolic
core and the future Python `ml/` subproject.

The contract is ownership-first:

- Go owns symbolic semantics and generated artifacts.
- Python consumes generated artifacts for learning, ranking, and assembly work.
- Python must not become a second symbolic engine with competing semantics.

Item 33 defines directory ownership, artifact families, and read/write
direction. Detailed schemas for new corpora are intentionally deferred to the
later TODO items that introduce them.

## Repository Layout

The repository is organized around three top-level concerns:

- Go code at repo root
  - parser, AST, concepts, evaluation, normalization, search, formal export,
    and experiment harnesses
- `experiments/`
  - oracle experiment specs and their generated run artifacts
- `artifacts/`
  - shared corpora intended to be generated by Go and consumed by `ml/`
- `ml/`
  - future Python-side training and evaluation code

The current Go code stays at repo root. Item 33 does not introduce a `/go`
subtree.

## Ownership Rules

### Go Owns

Go is the source of truth for:

- raw EML syntax and parsing,
- concept expansion,
- numeric evaluation semantics,
- normalization used for hygiene and controlled comparison,
- deterministic synthetic data generation,
- equivalence-family generation,
- experiment specs and experiment-run artifacts,
- shared corpora written under `artifacts/`.

### Python Owns

Python `ml/` will own:

- model training,
- model evaluation,
- representation learning over tree families,
- snippet discovery,
- equivalence-aware ranking or generation,
- assembly experiments over partial trees.

Python should read artifacts produced by Go rather than reimplementing symbolic
semantics ad hoc.

## Directory Responsibilities

### `experiments/`

This directory is reserved for the oracle experiment system.

- `experiments/specs/`
  - source-of-truth experiment inputs
- `experiments/datasets/`
  - generated oracle datasets
- `experiments/results/`
  - generated per-run outputs
- `experiments/reports/`
  - generated suite summaries

`experiments/` is not the general-purpose ML corpus area.

### `artifacts/`

This directory is reserved for shared corpora generated by Go and consumed by
other subsystems, especially `ml/`.

- `artifacts/equivalence/`
  - generated equivalence-family corpora
- `artifacts/snippets/`
  - generated snippet and partial-law corpora

Generated content here is reproducible and ignored by default unless there is
an explicit reason to commit a specific artifact later.

### `ml/`

This directory is reserved for the Python ML subproject.

Item 33 only reserves the location and documents the contract. It does not yet
choose a Python packaging tool, environment manager, or model framework.

## Artifact Families

The contract currently recognizes these artifact families.

### Oracle Experiment Artifacts

- experiment specs
- generated experiment datasets
- generated experiment results
- generated experiment reports

These remain under `experiments/`.

### Shared ML-Facing Corpora

- equivalence-family corpora
- snippet and partial-law corpora

These belong under `artifacts/`.

The equivalence-family corpus now has two layers:

- item-35 anchor artifacts
  - one generated anchor tree per file with sampled point sets
- item-36 family artifacts
  - one explicit equivalence family per file, including:
    - one anchor entry,
    - one or more related member entries,
    - explicit relation types,
    - shared deterministic sample bundles for later paired/grouped dataset generation

The current item-34/35 ML `family_match` path still reads the simpler anchor
artifacts. The richer family-level corpus is an additive artifact family for
later items rather than an in-place replacement.

The equivalence-family dataset layer now adds:

- `artifacts/equivalence/paired/`
  - grouped equivalence datasets derived from item-36 family artifacts
  - each file preserves:
    - family identity,
    - multiple sampling domains,
    - shared sampled point sets,
    - ordered anchor/member descriptors,
    - relation labels

These grouped datasets are the intended bridge from symbolic equivalence
families to later ML tasks that need paired or grouped examples rather than
only one anchor tree per file.

## Read / Write Direction

The intended data flow is:

- Go reads source-of-truth experiment specs
- Go writes generated oracle artifacts under `experiments/`
- Go writes shared corpora under `artifacts/`
- Python `ml/` reads from `experiments/` and `artifacts/`
- Python may write its own run-local outputs inside `ml/` or a later
  ML-specific output area, but item 33 does not define that yet

## Deliberately Deferred

Item 33 does not define:

- exact JSON schemas for equivalence-family corpora,
- exact JSON schemas for snippet corpora,
- Python environment or dependency tooling,
- ML training entrypoints,
- whether later ML outputs should live under `artifacts/`, `ml/`, or another
  dedicated top-level directory.

Those decisions belong to later TODO items:

- item 34 for Python subproject scaffolding,
- item 36 for equivalence-family corpus format,
- item 38 for snippet-level dataset generation.
